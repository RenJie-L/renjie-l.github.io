> 基于 [Cursor Agent 官方指南](https://cursor.com/cn/blog/agent-best-practices) 的总结。

## 一、了解 Agent Harness

Agent 建立在三个组件之上：

1. **Instructions**：引导 Agent 行为的 system prompt 和规则
2. **Tools**：文件编辑、代码库搜索、终端执行等工具
3. **User messages**：你用来指挥工作的提示词和后续交互

Cursor 会为每个模型专门调整 instructions 和工具。不同模型对同一提示的响应方式不同：有的更偏好 grep，有的需要明确要求在编辑后调用 linter。Harness 帮你处理这些差异，让你专注构建软件本身。

## 二、从规划开始

> 你能做出的最大改变：**在编写代码之前先进行规划**。

芝加哥大学研究发现，有经验的开发者更倾向于在生成代码之前先规划。规划让你更清晰自己要构建什么，并为 Agent 提供明确目标。

### 2.1 使用 Plan 模式

在 Agent 输入框按 **`Shift + Tab`** 切换 Plan 模式。Agent 不会立即写代码，而是：

1. 分析代码库以查找相关文件
2. 就你的需求提出澄清问题
3. 创建包含文件路径和代码引用的详细实现计划
4. 在开始构建前等待你的确认

计划以 Markdown 打开，可直接编辑、删减、补充。点击 “Save to workspace” 可保存到 `.cursor/plans/`，方便团队协作和中断后续写。

简单小改动或已经做过多次的任务不一定需要计划，可以直接交给 Agent。

### 2.2 从计划重新开始

若 Agent 生成的内容不符合预期，**不要反复修补提示**，而应：

1. 回滚改动
2. 重新细化计划，把需求描述得更具体
3. 再执行一次

这通常比修补一个进行中的 Agent 更快，结果也更干净。

## 三、管理上下文

随着 Agent 编写的代码越来越多，你的主要工作会变成：**为每个 Agent 提供完成任务所需的上下文**。

### 3.1 让 Agent 自行获取上下文

不必在提示里手动引用每一个文件。Agent 具备 grep 和语义搜索能力，会按需查找。例如询问 “the authentication flow” 时，Agent 会搜索到相关文件，即使提示里没有包含具体文件名。

原则很简单：

- 知道确切文件，就在提示中引用
- 不知道，就交给 Agent 去查
- 不要包含大量不相关文件，否则会模糊重点

### 3.2 善用引用

- **`@Branch`**：让 Agent 了解当前工作，例如 “Review the changes on this branch” 或 “What am I working on?”
- **`@Past Chats`**：新对话时引用历史。Agent 可以有选择地读取，只提取所需上下文，比复制整段对话更高效

![通过引用以往的聊天记录，将之前对话的上下文带入当前会话](https://cursor.com/marketing-static/_next/image?url=https://ptht05hbb1ssoooe.public.blob.vercel-storage.com/assets/blog/past-chats.jpg&w=1920&q=70)

### 3.3 何时开新对话，何时继续当前对话

| 开始新对话                 | 继续当前对话                |
| -------------------------- | --------------------------- |
| 切换到不同任务或功能       | 对同一功能迭代              |
| Agent 困惑、重复犯同样错误 | Agent 需要先前讨论的上下文  |
| 完成一个逻辑完整的工作单元 | 正在调试 Agent 刚构建的内容 |

过长对话会积累噪音，Agent 容易分散注意力。如果效果下降，应及时开新对话。

## 四、扩展 Agent：Rules 与 Skills

### 4.1 Rules：静态上下文

在 `.cursor/rules/` 创建 Markdown 规则，**每次对话都会加载**。规则应聚焦常用命令、工程模式和典型示例：

```markdown
# Commands

- `npm run build`: Build the project
- `npm run typecheck`: Run the typechecker
- `npm run test`: Run tests (prefer single test files for speed)

# 代码风格

- 使用 ES 模块，而非 CommonJS
- 尽可能使用解构导入：`import { foo } from 'bar'`
- 参考 `components/Button.tsx` 了解标准组件结构

# Workflow

- Always typecheck after making a series of code changes
- API routes go in `app/api/` following existing patterns
```

应避免复制整份风格指南、记录 Agent 已经了解的所有常用命令，或为极少出现的边缘情况增加复杂规则。从简单开始，只有 Agent 反复犯同样错误时再新增规则。将规则提交到 Git，团队就可以共享。

### 4.2 Skills：动态能力

Skills 在 Agent **认为相关时**才加载，可以包含：

- **Custom commands**：用 `/` 触发的可复用工作流
- **Hooks**：在 Agent 动作前后运行的脚本
- **Domain knowledge**：按需调用的任务指令

### 4.3 示例：长时间运行 Agent 循环

可以用 Hooks 让 Agent 持续迭代，直到达成测试全通过等目标。

`.cursor/hooks.json`：

```json
{
  "version": 1,
  "hooks": {
    "stop": [{ "command": "bun run .cursor/hooks/grind.ts" }]
  }
}
```

`.cursor/hooks/grind.ts` 从 stdin 接收上下文，并返回 `followup_message` 驱动下一轮：

```ts
import { existsSync, readFileSync } from 'fs';

interface StopHookInput {
  conversation_id: string;
  status: 'completed' | 'aborted' | 'error';
  loop_count: number;
}

const input: StopHookInput = await Bun.stdin.json();
const MAX_ITERATIONS = 5;

if (input.status !== 'completed' || input.loop_count >= MAX_ITERATIONS) {
  console.log(JSON.stringify({}));
  process.exit(0);
}

const scratchpad = existsSync('.cursor/scratchpad.md')
  ? readFileSync('.cursor/scratchpad.md', 'utf-8')
  : '';

if (scratchpad.includes('DONE')) {
  console.log(JSON.stringify({}));
} else {
  console.log(
    JSON.stringify({
      followup_message: `[迭代 ${input.loop_count + 1}/${MAX_ITERATIONS}] 继续工作。完成后在 .cursor/scratchpad.md 中更新为 DONE。`,
    }),
  );
}
```

这适合一直运行直到测试通过、反复迭代 UI 直到与设计稿一致，或任何目标可验证的任务。

## 五、常见工作流

### 5.1 测试驱动开发（TDD）

1. 让 Agent 根据**预期输入和输出**编写测试，明确说明正在做 TDD，避免它直接 mock 实现
2. 运行测试并确认失败，此时不写实现
3. 提交测试
4. 让 Agent 编写能通过测试的代码，不修改测试，并持续迭代到全部通过
5. 提交实现

Agent 有清晰迭代目标时表现最好，测试让它能够不断评估结果并改进。

### 5.2 理解代码库

像向同事请教一样提问，Agent 会使用搜索与语义理解寻找答案。例如：

- “这个项目里的日志是如何运作的？”
- “我该如何添加一个新的 API endpoint？”
- “`CustomerOnboardingFlow` 处理了哪些边界情况？”
- “为什么第 1738 行用的是 `setUser()` 而不是 `createUser()`？”

### 5.3 Git 工作流：自定义命令

将常用流程写成命令，放在 `.cursor/commands/`，用 `/` 触发。

```markdown
为当前更改创建 Pull Request。

1. 使用 `git diff` 查看已暂存和未暂存的更改
2. 根据更改内容编写清晰的提交信息
3. 提交并推送到当前分支
4. 使用 `gh pr create` 创建 Pull Request，并提供标题和描述
5. 完成后返回 PR URL
```

其他常用命令可以包括 `/fix-issue [number]`、`/review` 与 `/update-deps`。

## 六、其他能力

### 6.1 包含图片

- **从设计到代码**：粘贴设计稿，Agent 能识别布局、颜色和间距，也可以使用 Figma MCP
- **可视化调试**：对错误或异常 UI 截图，比文字描述更高效
- **浏览器**：Agent 可以控制浏览器截屏、测试应用并验证界面变化

### 6.2 代码审查

- **生成中**：实时查看 Diff，方向不对时按 Escape 中断
- **Agent Review**：完成后逐行分析并标记潜在问题
- **Bugbot**：推送后为 PR 自动做代码审查
- **架构图**：让 Agent 创建 Mermaid 图，说明复杂系统的数据流

### 6.3 并行运行 Agent

- Cursor 会为并行 Agent 自动创建 Git worktrees，各 Agent 隔离运行
- 同一提示可以同时运行多个模型，对比结果后选择最优方案
- 适合棘手问题、比较不同模型和发现边缘情况

### 6.4 云端 Agent

云端 Agent 适合“丢进待办”的任务，例如 bug 修复、重构、补测试和文档更新。基本流程是：描述任务，Agent 克隆仓库并创建分支，自主工作，完成后发起 PR，最后由你审查并合并。

### 6.5 Debug 模式

针对难以定位的 bug，Debug 模式会：

1. 生成多个可能出错原因的假设
2. 用日志语句埋点
3. 要求你在收集数据时复现 bug
4. 分析实际行为并定位根因
5. 基于证据做针对性修复

它适合能复现但难以解释的问题、竞争条件、性能问题和回归 bug。你描述得越具体，Agent 添加的埋点越有价值。

## 七、打造你的工作流

### 7.1 写具体的提示

指令越具体，成功率越高。

- 不够具体：“add tests for auth.ts”
- 更好的提示：“Write a test case for auth.ts covering the logout edge case, using the patterns in `__tests__/` and avoiding mocks.”

### 7.2 推荐与避免

**推荐**：

- 规划优先、写具体提示、认真 Review
- 提供可验证目标：强类型、linter、测试
- 把 Agent 当有能力的协作者：要计划、要解释，也允许它质疑方案

**避免**：

- 一次性塞入过多复杂需求
- 缺少项目背景和上下文
- 期望 AI 理解所有业务细节
- 在效果下降时仍坚持在长对话中修补
- 在真正理解自己的模式前过度优化规则
