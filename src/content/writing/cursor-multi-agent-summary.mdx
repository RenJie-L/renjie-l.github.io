# Cursor 多 Agent 自主编码研究

> 参考文档：
>
> - [扩展长时间运行的自主编码能力](https://cursor.com/cn/blog/scaling-agents)
> - [迈向自动驾驶代码库](https://cursor.com/cn/blog/self-driving-codebases)

---

## 一、研究背景与目标

### 1.1 核心问题

**单个 Agent 的局限**：当今的 Agent 在执行专注的小任务时表现不错，但在复杂项目上却显得缓慢。推进一个大型项目的路径往往并不明确，合理的工作拆分在一开始也并不清晰。

**研究目标**：让编码 Agent 持续运行数周乃至更长时间，完全自主工作，探索在通常需要人类团队耗时数月完成的项目上，Agent 驱动的编码前沿究竟能推进多远。

**核心假设**：能否通过向一个问题投入更多 Agent（多花 10 倍算力成本），换来 10 倍更有意义的吞吐量？

### 1.2 实证成果

| 指标            | 数据                                                  |
| --------------- | ----------------------------------------------------- |
| 并发 Agent 数量 | 数百个，峰值数百                                      |
| 单次运行代码量  | 超过 100 万行                                         |
| Token 消耗      | 数万亿                                                |
| 峰值吞吐        | 约每小时 1,000 次 commit（一周内 1,000 万次工具调用） |
| 运行时长        | 最长连续运行近一周，部分实验 3 周以上                 |

### 1.3 实际案例

- **从零构建浏览器**：1,000+ 文件，100 万行代码，[GitHub 源码](https://github.com/wilsonzlin/fastrender)
- **Cursor 代码库 Solid→React 迁移**：持续 3 周多，+266K/-193K 行代码，已通过 CI
- **视频渲染优化**：Rust 实现让速度提升 25 倍，已合并生产
- **Java LSP**：7.4K 次提交，55 万行代码
- **Windows 7 模拟器**：14.6K 次提交，120 万行代码
- **Excel**：12K 次提交，160 万行代码

![展示 Solid 到 React 迁移的 Pull Request](https://cursor.com/marketing-static/_next/image?url=https://ptht05hbb1ssoooe.public.blob.vercel-storage.com/assets/blog/long-running-agents-pr.png&w=1920&q=70)

### 1.4 技术起点

研究最初是个人业余项目，以「在不依赖 JavaScript 的情况下渲染网页」为目标。使用 Opus 4.5 写详细规划并不断提示「继续」——这很快失败：模型会搞不清自己在做什么、过早宣称完成、卡在实现细节上。但模型在小范围内展现出深厚的知识和智能。

**关键转折**：GPT-5.1/5.2 在精确遵循指令方面表现更好，团队将调度框架更新为使用 OpenAI 模型，框架得以构建简易无 JS 浏览器，但单 Agent 构建完整引擎仍慢到难以接受。

## 二、多 Agent 协作的演进历程

### 2.1 阶段一：自我协同（失败）

**设计思路**：让处于同一层级的各个 Agent 使用一个共享状态文件，查看其他 Agent 在做什么，决定自己要做什么，并将更新写回该文件。尽量少规定，让 Agent 自行摸索如何自我协作。

![自我协同：多个 Agent 连接到共享协调文件](https://cursor.com/marketing-static/_next/image?url=https://ptht05hbb1ssoooe.public.blob.vercel-storage.com/assets/blog/autonomy-diagram-1-3.png&w=1920&q=70)

**失败原因（详细）**：

1. **锁机制问题**
   - Agent 持有锁太久、忘记释放
   - 在不合法时尝试加锁/解锁
   - 不理解持有锁的意义
   - 锁机制既易用错又往往「勉强正确」，增加提示无济于事
   - 20 个 Agent 的有效吞吐量降到 1–3 个水平，大部分时间在等待锁
2. **系统脆弱**
   - Agent 可能在持有锁时失败
   - 尝试获取自己已持有的锁
   - 在完全未获取锁的情况下更新协调文件
3. **行为规避**
   - 缺乏结构化分工，没有任何单个 Agent 接手大型、复杂任务
   - 刻意回避争用和冲突
   - 选择小而安全的修改，而非对整个项目负起整体责任
   - 工作长时间空转，无实质性进展

**替代尝试**：乐观并发控制——Agent 可自由读取状态，若自上次读取后状态已变化则写入失败。更简单、更健壮，但更深层问题（缺乏层级、无人担责）依然存在。

---

### 2.2 阶段二：添加结构和角色（Planner + Executor + Judge）

**设计**：对角色进行划分，为各个 Agent 明确职责和责任归属。

![结构化角色：Planner、Executor、Workers、Judge 流水线](https://cursor.com/marketing-static/_next/image?url=https://ptht05hbb1ssoooe.public.blob.vercel-storage.com/assets/blog/autonomy-diagram-2-3.png&w=1920&q=70)

**角色定义**：

| 角色                   | 职责                                                        |
| ---------------------- | ----------------------------------------------------------- |
| **规划器（Planner）**  | 制定明确的方案和交付物，推动用户指令向前推进                |
| **执行者（Executor）** | 唯一主导 Agent，全权确保计划被完整落实；可派发任务给 Worker |
| **工作者（Workers）**  | 领取任务并专注完成，实现线性扩展和吞吐量提升                |
| **裁决者（Judge）**    | 执行者完成后介入，判断是否完成、是否需要下一轮迭代          |

**效果**：解决许多协同问题。将执行的所有权与统筹责任集中在单一角色，Worker 可专注于具体任务，系统稳定交付。

**观察与爬山优化**：

- 主要问题往往在多个 Agent 和工具调用中反复出现（如大量 `git restore` 导致资源争用）
- 使用 Cursor 分析日志，与提示词对比，理解行为与预期不符的原因
- **新问题**：瓶颈在最慢的 Worker；系统过于僵化；静态规划在发现新问题时难以动态调整；部分 Agent 朝错误方向前进，需等到下一轮迭代才能纠正

---

### 2.3 阶段三：持续执行器（移除独立 Planner）

**设计**：移除独立 Planner。执行器既生成任务，也规划如何实现目标。作为唯一 Agent，不需要把计划写到任何地方，也不必拘泥于静态计划，更不用死板等待所有 Worker 完成。

**新鲜度机制**（防止长时间运行后偏离预期）：

1. **scratchpad.md**：应频繁从头重写，而非不断追加
2. **上下文总结**：Agent 接近上下文长度上限时自动进行总结
3. **System prompt**：加入自我反思和目标对齐提醒
4. **灵活性**：鼓励 Agent 随时调整方向、质疑既有假设

**效果**：系统高度动态灵活，可主动探索代码、重新审视决策、管理 Worker、交错执行任务，持续反映最新信息。因 Agent 在按指令完成任务方面表现可靠，移除 Judge 以保持简单。

![持续执行器：包含多个 Worker 的无限执行循环](https://cursor.com/marketing-static/_next/image?url=https://ptht05hbb1ssoooe.public.blob.vercel-storage.com/assets/blog/autonomy-diagram-3-3.png&w=1920&q=70)

**异常行为**：

持续执行器被赋予过多角色后表现出病理行为：

- 随机休眠
- 停止运行 Agent
- 自己亲自做工作
- 拒绝规划，只生成少量极窄任务
- 无法正确合并 Worker 更改
- 过早声称工作完成

**原因分析**：同时承担规划、探索、研究、派生任务、检查 Worker、审查代码、执行编辑、合并输出、判断循环完成——角色过载导致被压垮。

---

### 2.4 阶段四：最终系统设计（递归规划 + Worker）

融合所有经验后的架构：

![最终系统设计：递归规划器、子规划器、Worker 与 Git](https://cursor.com/marketing-static/_next/image?url=https://ptht05hbb1ssoooe.public.blob.vercel-storage.com/assets/blog/autonomy-diagram-4-3.png&w=1920&q=70)

**三大角色**：

1. **根规划器（Root Planner）**
   - 负责用户指令的整体范围
   - 理解当前状态，给出具体、明确的任务，使工作朝目标推进
   - **不编写任何代码**
   - 不知道任务是否被领取、由谁执行
2. **子规划器（Sub-Planner，递归）**
   - 当规划器认为范围可进一步拆分时生成
   - 完全接手被委派的子范围，负全责
   - 可快速并行拉起 Worker 提升吞吐量
   - 避免单个规划器不堪重负、视野变窄
3. **Worker**
   - 领取任务并全程推动完成
   - 不了解更大系统，不与其他规划器或 Worker 通信
   - 在自有代码副本上工作
   - 完成后写出**交接说明**，由系统提交给发起任务的规划器

**交接说明内容**：已完成事项 + 重要备注、关注点、偏差、发现、想法、反馈。规划器将其作为后续消息接收。

**关键特性**：

- 信息沿链路向上传播，无需全局同步或跨 Agent 额外沟通
- 每个 Agent 全权「拥有」其范围
- 即使规划器已「完成」，仍持续接收更新、拉取最新仓库，可继续规划和决策
- 与当今某些软件团队的运作方式相呼应

**移除 Integrator**：

- 最初为集中式质量控制、减少 push/rebase/冲突/合并的资源竞争而引入
- 迅速成为瓶颈：数百 Worker 只有一道关卡
- 尝试调整 prompt 无效，最终移除；Worker 本身有能力处理冲突

## 三、吞吐量与权衡

### 3.1 提交正确性

**100% 正确要求的影响**：

- 导致严重串行化，有效吞吐大幅下降
- 即使小错误（API 变更、拼写）也会让整个系统几乎停摆
- Worker 越界修复不相干内容
- 多个 Agent 一拥而上，彼此「踩踏」同一问题

**策略**：允许一定误差空间。Agent 可相信其他问题会很快被其他 Agent 修复——实践中确实如此，因系统对代码库有有效的所有权和委派。错误会出现，但会快速修复；错误率保持较小且稳定，平稳可控。

**启示**：理想高效系统接受一定错误率，但需要最终「绿色」分支，由某 Agent 定期快照并在发布前做快速修复清理。

### 3.2 同步开销

**现象**：多个 Agent 同时修改同一文件或重构同一代码段。

**策略**：接受偶发「乱流」，让系统在短时间内自然收敛恢复稳定，而非完全杜绝或过度设计。

**代价**：多消耗 token，局部造成竞争。

**收益**：整体更简单——更容易对齐模型、避免压垮、易管理观测、阻力更小、全局生产效率更高。

## 四、基础设施经验

### 4.1 架构选择

- **单机优先**：每次多 Agent 运行在独立的大型 Linux VM 上，资源充足，避免过早引入分布式复杂性
- **可观测性**：前期投入记录所有 Agent 消息、系统操作、命令输出（附时间戳），便于分析、回放、筛选、发现模式

### 4.2 瓶颈与启示

- **RAM 限制后，磁盘成为瓶颈**：数百 Agent 同时编译，单体项目导致构建产物数 GB/s 级读写
- **教训**：项目结构、架构决策、开发者体验会影响 token 和 commit 吞吐——与代码库交互（如编译）占主导，而非理想中的思考和编码

### 4.3 未来方向

- **Git/Cargo 等工具**：使用共享锁做并发控制；可考虑引入数据库等并发系统中的成熟机制
- **写时复制与去重**：Agent 各有仓库副本，但大多文件相同；可借鉴生产级存储的简单特性，为「单用户」系统带来效率提升

## 五、向代理明确说明意图

### 5.1 指令的重要性

本质上是与典型编码 Agent 交互，但给予高出数个数量级的时间和算力——**会放大一切**，包括次优或含糊的指令。Agent 被训练为严格遵循指令，不会更改或推翻指令，即便指令本身很糟糕。

### 5.2 浏览器项目中的教训

| 问题                 | 原因                            | 纠正方式                |
| -------------------- | ------------------------------- | ----------------------- |
| 深挖晦涩、极少用特性 | 「spec implementation」过于含糊 | 明确优先级和范围        |
| 性能超出可接受范围   | 隐含假定                        | 明确指令 + 强制超时配置 |
| 内存泄漏、死锁       | 复杂部分需显式资源管理          | 基于进程的资源管理工具  |
| 架构不适于演化       | 初始规格设计失败                | 重新设计规格            |
| 引入本可自实现的依赖 | 指令疏忽                        | 明确依赖理念及禁止库    |

**正面案例**：明确依赖理念后，系统进行大规模重构，拆分为自包含 crate，从单体脱离。仓库一度严重损坏，多 Agent 系统在几天内收敛到可工作代码，等待编译时间大幅减少，吞吐量为之前数倍。

## 六、优化提示词

### 6.1 核心原则

1. **只指令「不知道的事」**：多 Agent 协作、代码库流程、部署流水线。把模型当成极聪明但刚入职的工程师——懂工程，不熟悉你的代码库和流程。
2. **约束优于指令**：如 `"No TODOs, no partial implementations"` 优于 `"remember to finish implementations"`。模型默认常做正确的事，约束是划定边界。
3. **避免打勾清单**：给出很多具体要做的事项，会让模型过于专注这些点而忽略更广范围，并隐含降低未列事项的优先级。更好的是让模型自主判断。
4. **具体数量有效**：`"generate many tasks"` 往往只生成少量（保守默认）；`"Generate 20-100 tasks"` 传达更大范围、更有野心，会观察到明显不同、更广泛的行为。

## 七、模型选择

- **长时间自主工作**：GPT-5.2 系列更优——更遵循指令、保持专注、避免偏离、实现更精确完整
- **Opus 4.5**：往往更早结束、走捷径、更快交还控制权
- **角色差异化**：GPT-5.2 作为规划者优于专为编码训练的 GPT-5.1-codex
- **实践**：针对每个角色选择最适合的模型，而非单一通用模型

---

## 八、系统设计经验总结

1. **反脆弱性**：随 Agent 数量增加，失败概率上升；系统需承受单个 Agent 失败，允许其他 Agent 恢复或尝试替代方案
2. **实证为先**：基于数据和观察调整，而非预设「系统应如何工作」的假设（无论来自人类组织还是现有系统设计）
3. **明确为吞吐量设计**：在编码其他方面做取舍，接受较小且稳定的错误率，通过最终核对修正，而非追求 100% 完美——那会显著拖慢系统
4. **简洁优于复杂**：正确设计时往往优雅简洁；在探索多种方案前，不清楚哪一种简单方法可行；许多改进来自「减法」而非「加法」

## 九、可观测性

- 记录所有 Agent 消息、系统操作、命令输出，附带时间戳
- 便于人工审查、回放会话，以及将数据回传给 Cursor 做模式发现
- 主要问题往往在多个 Agent 和工具调用中反复出现

## 十、对个人/小团队的启示

| 场景       | 可借鉴做法                                               |
| ---------- | -------------------------------------------------------- |
| 任务拆解   | 大任务 → 递归拆分为子任务 → 明确交付物与交接说明         |
| 角色划分   | 规划（不写代码）与执行分离，避免单一 Agent 负担过重      |
| 上下文管理 | 定期重写 scratchpad、接近上限时总结                      |
| 指令设计   | 具体数量、明确约束、避免含糊的「spec implementation」    |
| 错误容忍   | 接受小错误率，通过迭代收敛，而非追求一次完美             |
| 结构程度   | 介于无结构（冲突、重复）与过度结构（脆弱）之间           |
| 减法思维   | 移除不必要的角色（如 Integrator、Judge）有时比增加更有用 |
