# 向智能体式 AI 的转变：来自 Codex 的证据

> 原论文：[**The Shift to Agentic AI: Evidence from Codex**](https://cdn.openai.com/pdf/5d1e1489-21c0-43e4-9d42-f87efdbf0082/the-shift-to-agentic-ai-evidence-from-codex.pdf)

## 摘要

这篇论文研究的重点不是“Codex 写代码有多强”，而是：当 AI 不再只是回答问题，而是能够读取文件、调用工具、运行命令、修改代码并持续执行任务时，人的工作方式会发生什么变化。

作者分析了 Codex 的真实使用数据，对比了三类人群：个人用户、企业和组织用户，以及 OpenAI 内部员工。

论文的核心结论是：

> AI 的主要交互单位，正在从“一次对话”变成“一项被委派的工作”。

用户不再只是让 AI 解释问题、提供建议或生成一小段内容，而是开始把完整任务交给它，包括理解环境、执行操作、验证结果和持续修正。

这种变化主要表现为：Codex 用户快速增长；使用场景由开发者向非开发者扩展；用户委派的任务越来越复杂；重度用户开始同时管理多个智能体；重复流程被整理为 Skills 和 Plugins；人的角色逐渐从直接执行者转向任务设计者、监督者和整合者。

---

# 1. 什么是 Agentic AI？

传统聊天式 AI 更像顾问：

```text
用户：这个报错是什么原因？
AI：可能是依赖版本不兼容，你可以尝试……
```

智能体式 AI 更像执行者：

```text
用户：检查整个项目，找到报错原因，修改代码，运行测试并整理修改说明。
AI：读取仓库 → 检查依赖 → 修改文件 → 执行命令 → 分析结果 → 继续修复
```

两者的核心区别，不是模型会不会生成文字，而是 AI 能否读取真实项目环境、调用终端或浏览器、执行连续的多步骤任务、修改实际工作产物，并在较少人工干预下持续推进工作。

论文将 ChatGPT 主要视为对话式 AI，将 Codex 主要视为智能体式 AI。两者并非绝对分开，但工具调用比例能反映差异。

截至 2026 年 6 月的一周内：

- 约 **60.3% 的 Codex 交互轮次**调用了至少一种工具；
- ChatGPT 中对应比例约为 **21.9%**。

软件开发最适合率先采用这种模式，因为代码和项目文件都是数字化的，而且可以通过编译、测试和代码审查验证结果。但论文也发现，Codex 已被用于研究、写作、数据分析、规划、招聘、销售和沟通等非编程任务。

---

# 2. 研究对象与方法

论文比较了三类用户。

## 个人用户

包括 Free、Go、Plus 和 Pro 用户。这个群体代表大众市场中的早期采用者，通常缺少企业内部系统、统一流程和组织培训。

## 组织用户

包括 Business 和 Enterprise 用户。企业采用 Codex 会受到权限、安全、合规、培训、管理推动和工作流程设计等因素影响。

## OpenAI 内部员工

OpenAI 内部被视为“低采用摩擦”的前沿环境：员工熟悉前沿模型，使用限制较少，组织内部高度支持 Codex，并存在培训、经验分享和内部工作流建设。

作者强调，OpenAI 内部不能代表普通企业平均情况，更像是展示未来可能形态的窗口。

研究并非由人工直接阅读大量用户对话，而是通过自动分类器提取聚合指标，包括用户类型、任务类别、任务复杂度、Skill 使用情况、并发数量和智能体运行时长。

这种方法适合分析大规模真实行为，但也意味着部分结论来自模型估计，而不是人工逐条核验。

---

# 3. 核心发现一：采用速度快，但非常不均衡

2026 年前六个月，Codex 周活跃用户增长超过 **5 倍**。

截至 2026 年 6 月附近的 28 天窗口：

| 用户类型    | 活跃用户中使用 Codex 的比例 | Codex 占 ChatGPT + Codex 输出 Token 的比例 |
| ----------- | --------------------------: | -----------------------------------------: |
| 个人用户    |                        0.7% |                                      16.5% |
| 组织用户    |                       17.3% |                                      63.3% |
| OpenAI 员工 |                       97.9% |                                      99.8% |

这组数据说明两个问题。

第一，个人用户中真正开始使用 Codex 的人仍然很少，Agentic AI 尚未全面普及。

第二，一旦开始使用，使用强度往往很高。虽然只有 0.7% 的个人活跃用户使用 Codex，但 Codex 已贡献个人用户总输出 Token 的 16.5%。

组织用户也存在类似情况：不是所有员工都在使用，但已经采用的人往往把大量工作转移到 Codex 中。

OpenAI 内部则几乎完成了从传统聊天工具到 Codex 工作流的迁移。不过这更多反映低使用摩擦、高熟练度和强组织支持下的前沿状态。

---

# 4. 核心发现二：Codex 主要用于“做事”，而不是“问问题”

传统聊天机器人常见的使用方式是获取信息、请求解释、学习概念、获得建议和进行讨论。

Codex 更常被用于修复错误、重构代码、配置环境、运行测试、管理仓库、生成文档、分析数据和修改实际工作产物。

例如：

```text
解释一下 Astro 客户端路由是什么。
```

这是知识咨询。

而：

```text
检查这个 Astro 项目，启用客户端路由，
修复页面状态问题，运行构建测试，并更新项目文档。
```

这是任务委派。

因此，Codex 输出 Token 的增加不只是“AI 说了更多话”，而是 AI 开始完成更长的工作链。

---

# 5. 核心发现三：软件开发仍是核心，但用途正在扩展

论文将 Codex 请求划分为多个大类：工程运维、代码实现、知识产物、代码理解、应用管理、代码验证、研究、协作、数据分析和业务流程。

不同群体的任务结构不同：

| 任务类别 | OpenAI | 组织用户 | 个人用户 |
| -------- | -----: | -------: | -------: |
| 工程运维 |    31% |      23% |      23% |
| 代码实现 |   3.7% |      19% |      24% |
| 知识产物 |   8.4% |      11% |      14% |
| 代码理解 |   5.3% |      15% |      12% |
| 应用管理 |    16% |     7.6% |     8.2% |
| 代码验证 |   7.6% |      10% |     5.9% |
| 研究     |   8.4% |     5.7% |     6.2% |
| 协作     |    12% |     3.7% |     2.9% |
| 数据分析 |   4.0% |     2.7% |     2.2% |
| 业务流程 |   3.1% |     0.9% |     1.0% |

个人用户最常让 Codex 直接写代码，组织用户更多用于开发、理解和维护现有系统。

OpenAI 内部则较少把 Codex 只当“代码生成器”，而是更多用于工程运维、应用管理、研究和协作。

这反映出一种可能的成熟路径：

```text
生成代码
→ 理解已有项目
→ 修复与测试
→ 配置环境
→ 管理应用和仓库
→ 生成文档
→ 支持研究、数据和业务流程
```

随着采用程度提高，Codex 会从单点工具逐渐进入完整生产流程。

---

# 6. 非开发者也在快速进入

Codex 最早由开发者采用，但论文发现，2026 年非开发者用户增长更快。

非开发者使用场景包括文档和写作、数据分析、研究、项目规划、招聘、销售、法务和内部沟通。

企业中，技术岗位仍明显领先：

- 工程岗位平均约 **26.8%** 的 AI 输出来自 Codex；
- 数据与分析岗位约 **15.2%**；
- 法务岗位约 **1.9%**。

但平均值会掩盖高度不均衡的使用分布。虽然平均工程师只有 26.8% 的 AI 输出来自 Codex，但从整个工程群体的 Token 总量看，Codex 已占 **88.3%**。

这意味着，一部分工程师已经非常重度地使用，而另一部分尚未开始。

OpenAI 内部的扩散也展示了类似路径：先由工程团队使用，然后扩展到法务、招聘等非技术部门。内部培训、经验分享和组织推动明显加快了采用速度。

---

# 7. 用户正在委派越来越复杂的任务

研究通过分类器估算：如果没有 AI，一个有经验的人完成某条请求大概需要多长时间。

这个数字不是实际计时，而是用于观察任务规模变化的近似指标。

个人 Codex 用户中，至少提交过一次“预计需要人类超过 8 小时完成”的任务的用户比例：

- 年初约为 **2.1%**；
- 后来上升到 **25.6%**。

这不代表 Codex 一定能完整、正确地完成每一个八小时任务，而是说明用户愿意尝试委派的工作边界迅速扩大。

过去常见请求可能是：

```text
修复这个函数。
```

现在更可能是：

```text
阅读整个仓库，理解认证流程，重新设计权限系统，
迁移相关接口，补充测试并整理迁移文档。
```

论文还发现，大任务通常出现在对话的第一条消息中，后续消息主要用于反馈和修正。

典型结构更像：

```text
总体委派
→ AI 执行
→ 人工审查
→ 局部修正
→ 最终验证
```

这说明智能体交互正在接近真实工作中的任务分配和阶段验收。

---

# 8. 并行智能体：人开始管理“AI 团队”

Codex 允许用户在多个线程中同时运行不同任务，例如：

```text
Agent A：修复后端错误
Agent B：重构前端
Agent C：补充测试
Agent D：整理文档
Agent E：研究另一套实现方案
```

在外部用户中，大多数人仍是单线程使用：

- 约 **63.9% 的个人用户**没有使用并发；
- 约 **67.4% 的组织用户**没有使用并发。

OpenAI 内部则明显不同：

- 只有 **10.7%** 的用户始终只运行单一工作流；
- **28.6%** 的用户曾同时管理 5 个或更多智能体。

当多个任务并行后，人的主要工作会转向拆分任务、分配优先级、提供上下文、监控状态、处理依赖、审查产出和整合结果。

这时，人不再只是使用一个 AI 工具，而是在管理一组 AI 工作流。

---

# 9. 长时间运行：智能体开始承担持续工作

截至 2026 年 6 月，OpenAI 内部员工每日 Codex 任务累计运行时长中位数约为 **2.5 小时**。

部分头部用户的智能体累计运行时长甚至超过一天。第 99 百分位用户平均每天累计运行约 **71 小时**的 Agent 任务。

这并不是一天真的有 71 小时，而是多个任务同时运行，累计时长相加后超过 24 小时。

这说明典型用户仍以间歇式使用为主，但一小部分重度用户已经开始建立持续、并行的智能体工作系统。

# 10. Skills 与 Plugins：把重复使用变成固定流程

临时使用 Codex 时，用户每次都需要重新说明项目背景、文件结构、命令和工具、输出格式、团队规范和验收标准。

Skills 和 Plugins 的作用，就是把这些重复信息固化下来。

## Skill

Skill 是面向某类任务的可复用工作流，通常包含：

```text
SKILL.md
脚本
参考资料
模板
资源文件
```

它定义什么时候使用、如何执行、允许调用哪些工具、遵循哪些规范，以及最终结果应满足什么标准。

## Plugin

Plugin 是更大的能力包，可以包含多个 Skills、应用集成、MCP 配置、Hooks 和支持资源。

可以简单理解为：

> Skill 定义如何完成任务；Plugin 负责打包和分发能力。

截至 2026 年 6 月：

| 用户类型        | 至少使用过一个 Skill 的比例 |
| --------------- | --------------------------: |
| 个人 Codex 用户 |                       25.7% |
| 组织 Codex 用户 |                       30.4% |
| OpenAI 内部用户 |                       96.2% |

整体 Skill 使用率从 2026 年 3 月的 **5.4%** 增长到 6 月的 **26.6%**。

这说明成熟用户不再满足于每次临时写提示词，而是开始把稳定流程整理成可复用能力。

对于企业，自定义 Skill 尤其重要，因为它可以保存内部 API 使用方式、发布流程、安全规则、写作格式、数据口径、代码审查标准和团队专属术语。

Skill 的价值不只是减少提示词，而是把组织经验转化为可复用、可审查、可版本管理的工作流程。

# 11. 对生产率和工作的影响

论文发现，OpenAI 内部不同岗位的 AI 输出规模都大幅增长。

从 2025 年 11 月到 2026 年 6 月：

- 法务岗位月度 AI 输出中位数约增长 **13 倍**；
- 研究岗位增长超过 **50 倍**。

但 Token 增长不等于生产率同比例增长。更多输出也可能包含中间过程、重试、修正和未采用结果。

这些数字更可靠地说明：

> AI 介入工作的规模和深度大幅上升。

传统 AI 通常只改善某一个环节：

```text
人工作
→ 问 AI
→ 人继续执行
```

Agentic AI 则可能完成连续工作链：

```text
读取资料
→ 制定方案
→ 修改文件
→ 运行验证
→ 修复问题
→ 生成报告
```

当 AI 能承担连续步骤时，人工反复接手的成本可能下降，因此它对生产率的潜在影响可能大于单点辅助工具。

---

# 12. 组织需要围绕 AI 重构流程

论文用制造业“电气化”作类比。

早期工厂只是用电动机替换蒸汽机，但没有改变生产布局，因此收益有限。真正的生产率提升发生在工厂围绕电力重新设计生产线之后。

Agentic AI 也可能经历类似阶段。

## 第一阶段：替换单个任务

```text
原来人工写一段代码
→ 现在让 AI 写
```

## 第二阶段：自动化完整流程

```text
AI 读取项目
→ 实现功能
→ 运行测试
→ 修复问题
→ 更新文档
```

## 第三阶段：重新设计组织

```text
一人管理多个智能体
→ Skills 固化流程
→ 权限、审核和责任重新设计
→ 团队结构与项目节奏变化
```

多数外部企业目前仍处于第一到第二阶段，而 OpenAI 内部部分团队已开始接近第三阶段。

真正的价值因此不只是“买一个更强模型”，而是重构工作拆分方式、系统权限、自动测试、审核流程、责任边界、培训体系和团队协作模式。

# 13. 人的角色会怎样变化？

随着 AI 承担更完整的任务，人类的工作重心可能转向：

- 确定目标；
- 拆分任务；
- 准备上下文；
- 设置约束；
- 指定验收标准；
- 审查结果；
- 协调多个智能体；
- 对最终结果负责。

因此，未来重要能力可能包括清楚描述目标、判断哪些任务适合委派、识别 AI 错误、验证事实和质量，以及协调并行工作流。

领域专家未必会变得不重要。相反，AI 能执行的任务越大，错误影响越大，专业审查和最终判断就越重要。

同时，初级岗位的成长路径可能被改变。许多新人过去通过写基础代码、整理数据和起草文档积累经验，如果这些任务被大量自动化，企业就必须重新设计培训和职业发展路径。

# 14. 论文的主要局限

这篇论文提供了重要的早期证据，但不能过度解读。

## OpenAI 内部不具代表性

OpenAI 员工熟悉 AI、使用限制少、组织支持强，因此不能直接推断普通企业会达到同样采用水平。

## 观察性研究不能证明因果

论文看到 Codex 使用增加、任务变复杂和输出增长，但不能严格证明生产率因此同比例提高。

## Token 不等于业务价值

生成更多内容不代表代码已经上线、报告已经采用或业务结果已经改善。

## 工时估计来自分类器

“需要人类 8 小时”的判断是模型估计，不是实际计时。

## 并发不一定提高质量

多个智能体同时工作也可能造成冲突、重复、错误传播和审查负担。

因此，论文更适合用来理解使用模式变化，而不是证明 Agentic AI 已经全面提升企业绩效。

# 15. 对个人开发者的启示

## 不要只把 Codex 当代码补全

低阶使用：

```text
帮我写一个 Astro 组件。
```

更成熟的使用：

```text
分析现有 Astro 项目，为博客增加访问统计功能。
要求不影响静态生成，完成组件、样式、配置、
构建验证和 README 更新。
```

## 给出完整任务结构

一个适合智能体执行的请求，最好包含：

```md
## 目标

需要实现什么结果。

## 当前环境

框架、目录、依赖和限制。

## 允许修改

可以改哪些文件。

## 执行要求

需要运行哪些测试和构建。

## 验收标准

什么状态算完成。

## 输出

需要提交哪些说明。
```

## 把高频流程写成 Skill

例如个人博客可以固化 MDX 文章整理、SEO 检查、图片本地化、构建错误排查、发布前检查和 Git 提交规范。

## 保留人工验收

至少检查构建是否成功、测试是否通过、是否引入安全问题、是否修改了不应修改的文件，以及事实和引用是否可靠。

---

# 16. 最终结论

这篇论文最重要的观点是：

> AI 正从“对话工具”转变为“可被委派工作的执行系统”。

成熟的 Codex 使用方式不是不停聊天，而是：

1. 给出明确目标；
2. 提供真实环境和上下文；
3. 允许智能体使用工具执行多步骤任务；
4. 并行运行多个相对独立的工作流；
5. 用 Skills 和 Plugins 固化重复流程；
6. 由人负责监督、验证和整合。

论文还不能证明 Agentic AI 已经在所有组织中显著提高生产率，但它清楚展示了一种正在形成的工作模式：用户开始委派更大的任务；智能体运行时间越来越长；重度用户管理多个并行任务；工作流被系统化和复用；人的角色从亲自执行转向任务设计与质量控制。

因此，这篇论文真正讨论的并不是 Codex 本身，而是未来工作的组织方式：

> **从“人使用 AI 完成任务”，转向“人设计并管理由 AI 执行的工作系统”。**
